# Reasonix Cognitive Architecture Guide v3.0

## 从 Reasonix 2B 系统生长出的认知架构完全指南

> 版本：v3.0 · 2026-06-02
> 创作方式：基于 Reasonix 2B 体系的真实运行经验反向推导 + 正向设计
> 适用对象：任何希望从"LLM 应用"进化为"认知操作系统"的 AI 系统

### 本文档的两种读法

如果你正在设计新系统，按目录顺序通读——这是一份完整的架构蓝图。

如果你是 Reasonix 2B 的维护者，每章末尾有一段 **「2B 当前状态」** ，标注你的系统现在处于什么位置、下一站是什么。

# 第零部分：元框架

## 0.1 第一原则

不要追求更聪明。

追求：**更少遗忘**。

绝大多数 AI 系统失败的原因不是推理能力不足。

而是：**今天学到的东西，明天消失了。**

2B 系统的三条宪法恰好是对此最精确的表达：

> 1. 永不伪造
> 2. 永不猜测
> 3. 永不跳过关键验证

用否定句式来约束肯定冲动——这正是"不遗忘"这一抽象原则的可操作版本。

## 0.2 能力公式

```
认知能力 = 推理能力 × 记忆能力 × 演化能力
```

三项中任意一项为零，乘积为零。

大多数 LLM 应用死于演化能力为零——每次对话都是独立的，没有跨会话积累。

## 0.3 大脑 vs 进化工厂

| 大脑模式 | 进化工厂模式 |
|---------|-------------|
| 记住答案 | 沉淀方法 |
| 存储事实 | 发现模式 |
| 被动接收 | 主动演化 |
| 单次对话 | 跨会话积累 |
| 上下文依赖 | 上下文无关 |

2B 的 `evolve` Skill 已经实现了从"大脑"到"工厂"的第一步——在每次会话结束时沉淀资产。但目前的 evolve 仍然是被动触发（会话结束/用户主动），还不是主动探测。

**下一站**：让 evolve 从"定时执行"进化为"事件驱动"。

## 0.4 系统的终极目标

> 下一次遇到相同问题时，比上一次做得更好。

如果一个系统无法利用过去改善未来，那么它只是拥有记忆，不是拥有智慧。

# 第一部分：记忆架构

## 1.1 分层组织（替代平面堆积）

**反模式**：所有内容塞进一个 MEMORY.md，5000 行、10000 行。

结果是上下文污染、检索困难、推理退化。

**正确设计**：每个文件职责单一，不混合。

2B 系统目前的 `memory/` 布局：

```
D:\Reasonix\memory\
├── MEMORY.md              ← 索引（当前 15 行，健康 ✅）
├── user_preferences.md    ← 用户偏好（简洁、TTS、命名习惯）
├── project_state.md       ← 当前架构、MCP、进行中决策
├── knowledge.md           ← 技术经验、踩坑、解决方案
├── lessons.md             ← 被纠正、失败、反模式
└── ...（session_20260602.md 等）
```

这是正确的起点。随着时间推移应自然分化为：

```
memory/
├── MEMORY.md                   ← 索引（控制在 200 行以内）
├── profiles/                   ← 用户画像与长期偏好
│   ├── user.md                 ← 审计部工程管理背景、备考一造
│   ├── preferences.md          ← 回复风格、TTS 引擎优先级、命名规范
│   └── workflow_prefs.md       ← 工具偏好、Git 工作流
├── decisions/                  ← 所有决策记录
│   └── decisions.md            ← 决策日志（含假设与预期）
├── knowledge/                  ← 事实型知识
│   ├── domain.md               ← 审计领域知识
│   ├── tech_stack.md           ← Python/MCP/Git/Tailscale
│   └── references.md           ← 外部资源指针
├── lessons/                    ← 错误与教训
│   ├── errors.md               ← 已发生的错误
│   └── near_misses.md          ← 几乎发生的错误
├── patterns/                   ← 重复规律
│   ├── active.md               ← 当前模式候选项
│   └── archived.md             ← 已固化为 Skill
├── skills/                     ← 固化能力
│   ├── evolve.md               ← 演化闭环
│   ├── sync.md                 ← 同步
│   ├── note_process.md         ← 知识整理
│   └── proposals/              ← 候选 Skill
└── meta/                       ← 关于记忆的记忆
    ├── changelog.md            ← 变更日志
    ├── health.md               ← 健康指标
    └── gaps.md                 ← 知识空白清单
```

## 1.2 每条记忆的生命周期

2B 目前的 memory 是"写入即确认"。更好的做法：

```
Hypothesis  →  Tentative  →  Confirmed  →  Stable  →  Archived
   │              │              │            │            │
   │ 单次观察     │ 2+次重复     │ 验证通过    │ 长期有效    │ 已过时
   └──────────────┴──────────────┴─────────────┴─────────────┘
```

每个 memory 文件中的条目都标注 `status`：

```yaml
status: tentative       # hypothesis | tentative | confirmed | stable | archived
confidence: 0.7         # 0.0 ~ 1.0
source_type: inference  # user_explicit | user_behavior | self_derived | external
observed_count: 3       # 支持该条目的观察次数
```

## 1.3 关于 MEMORY.md 的健康红线

2B 的 evolve Skill 已经有一条重要规则：
> 检查 MEMORY.md 行数健康度（>200 行立即拆分）

这条规则值得推广到所有索引文件。200 行是上限，不是目标。目标是把每行控制在 80 字符以内，只保留指针，不保留正文。

# 第二部分：记录 Delta，而非 Snapshot

## 2.1 增量原则

**错误**：每天写日志式的快照。

```
[2026-06-01] 讨论了 MCP 配置
[2026-06-02] 讨论了 Voice Server
```

**正确**：记录变化本身。

```
[2026-06-02] 新增：voice_server.py 路径从 D:\AgentWork 迁移到 scripts/lib/
[2026-06-02] 修正：CLAUDE.md 从 30 行扩展至 58 行
[2026-06-02] 废弃：AgentWork 作为主工作区的角色 → 转为历史归档
```

## 2.2 三种值得记录的变化

| 操作 | 含义 | 2B 实例 |
|------|------|---------|
| 新增 | 从未知到已知 | 新增 MCP voice 服务器的配置记录 |
| 修正 | 从错误到正确 | voice_server.py 硬编码路径修正 |
| 废弃 | 从有效到无效 | AgentWork 降级为历史归档 |

写入前先问：这是新增、修正、还是废弃？答不出来就别写。

## 2.3 变更日志

每个 memory 文件尾部追加 changelog，或统一写入 `meta/changelog.md`。

```
- 2026-06-02: 新增 knowledge.md - MCP 配置要点
- 2026-05-28: 修正 project_state.md - 当前活跃 MCP 列表更新
- 2026-05-20: 废弃 project_state.md - old_github_mcp 条目（已切换到 gh CLI）
```

# 第三部分：知识与决策分离

## 3.1 混在一起是万恶之源

> 2B 现状：目前 knowledge.md 和 project_state.md 已有初步分工。但 decisions.md 尚未建立。

**知识** = 事实。
**决策** = 选择。

知识是客观的，决策是主观的。知识可以被推翻，决策可以被逆转。它们需要不同的评估标准和不同的存储位置。

**反例观察**：当用户问"为什么用 gh CLI 而不是 GitHub MCP？"时，当前系统能回答"因为 decision.md 里记录过"。但如果这个决策没有显式记录，系统就只能猜或做出不稳定的推断。

## 3.2 决策记录模板（建议 2B 立即采用）

```yaml
---
id: D-2026-0602-001
status: active           # active | superseded | reversed
area: mcp               # 决策领域
---
## 决策
采用 gh CLI 而非 GitHub MCP 作为 Git 操作接口

## 背景
需要 GitHub 操作能力，有两个候选

## 候选方案
1. GitHub MCP 服务器 — 优点：原生工具集成；缺点：额外配置、依赖 npx
2. gh CLI — 优点：已预装、权限简洁、命令行可脚本化；缺点：需 shell 权限配置

## 最终选择
gh CLI（当前）

## 原因
1. 减少 MCP 服务器数量（voice + filesystem 已够）
2. gh CLI 在 shellAllowed 中已有配置
3. MCP 协议变更时 gh CLI 不受影响

## 假设
- gh CLI 的权限模型满足所有需求
- 未来不需要 GraphQL 级操作

## 预期结果
操作延迟略高于 MCP，但稳定性和可维护性更高

## 后续记录
[待回填实际效果]
```

## 3.3 被否决方案的独特价值

> 2B 的 lessons.md 已经在记录错误。建议扩展：在 decisions/ 中同时记录"否决了什么"。

被否决的方案是决策记录的半条命。没有它，后人无法理解"为什么选了A而不是B"——他们只知道选了A，不知道 B 曾经在桌面上。

## 3.4 When Decision Becomes Knowledge

当一个决策被反复验证、不再需要重新讨论时：

1. 核心结论摘要进入 `knowledge/`
2. 完整决策链保留在 `decisions/`（含否决方案）——**不可删除**
3. `decisions/` 中的条目标记为 `archived` 状态

# 第四部分：错误库优于知识库

## 4.1 不对称价值

一条失败记录的价值大于十条成功经验。

成功经验有幸存者偏差、归因困难、难以复制。
但失败记录是一条完整的因果链：**问题 → 原因 → 修正 → 规则**。

## 4.2 三阶错误防御

这是从 2B 的"自愈协议"中提炼的通用模式：

| 层级 | 触发条件 | 动作 | 状态 |
|------|---------|------|------|
| 一阶 | "停/不对" | 立即中断，保留上下文重来 | 已实施 |
| 二阶 | 两次修复失败 | 停止，输出错误分析，请求干预 | 已实施 |
| 三阶 | 两次纠正仍不对 | 清空上下文重来 | 已实施 |

这其实是"错误库"的运行时版本——不是事后记录，而是实时防御。

## 4.3 错误记录的完整模板

```
---
id: L-2026-0602-001
severity: high          # critical | high | medium | low
category: deployment    # 分类
status: resolved        # open | investigating | resolved | verified
---
## 问题
全局 config 的 shellAllowed 安全边界过宽（`del` `reg` `wmic` `C:\`）

## 原因
迁移时未区分全局 config 和项目 config 的权限范围

## 修复
审查并收紧 shellAllowed，移除 `del` `reg` `wmic` 等危险指令

## 防止复发规则
每次 config 变更时审计一次 shellAllowed 风险条目

## 类似场景
- 任何权限配置的复制粘贴操作
- 从旧系统迁移配置到新系统

## 验证
[待回填]
```

## 4.4 错误分类体系

```
ERROR_SECURITY    — 权限/安全相关（shellAllowed 过宽）
ERROR_PATH        — 路径/文件位置相关（硬编码 AgentWork）
ERROR_CONFIG      — 配置相关（MCP 注册、环境变量）
ERROR_DEPLOYMENT  — 部署/环境相关（workspace 迁移）
ERROR_TIMING      — 时效性判断
ERROR_SCOPE       — 范围误判
```

当某个分类出现 3 次以上，说明这里有系统性缺陷，需要写入 `patterns/` 作为警戒模式。

## 4.5 Near-Miss 记录

记录"几乎犯了但没犯"的错误：

```
---
type: near-miss
id: NM-2026-0602-001
---
## 场景
准备在全局 config 中注册新的 MCP 服务器时，
注意到 filesystem 已被注册，改为在项目 config 中注册。

## 如果未避免
会在全局 config 中累积不必要的 MCP 配置，造成污染。

## 触发条件
当新增 MCP 服务器时，先检查是否应在项目级而非全局级注册。

## 规则
MCP 服务器优先注册在项目级；只有工具服务器才注册在全局级。
```

# 第五部分：模式发现系统

## 5.1 2B 当前的状态缺口

2B 的 evolve Skill 已经在做三类资产提取：知识、纠错、决策。但它还没有 **模式发现** 这一层。

> evolve 总结本周 → 未来的 evolve 应该能说"本周发现了一个模式：你第 3 次在处理 PDF 相关任务了，可能需要一个 OCR Skill"

## 5.2 知识 vs 模式

| | 知识 | 模式 |
|--|------|------|
| 定义 | 孤立事实 | 关联结构 |
| 2B 实例 | "用户喜欢 Markdown" | "用户所有可复用内容偏向 Markdown，说明重视版本控制和纯文本可移植性" |
| 2B 实例 | "用户经常整理 PDF" | "用户存在稳定的 OCR 工作流，可 Skill 化" |
| 输出 | 可供引用 | 可产生 Skill |

## 5.3 2B 中的模式发现时机

从 2B 现有的三个 Skill 来看，模式发现的最佳触发点：

```
evolve（会话结束）→ 扫描对话 → 发现同类任务出现 3+ 次
                    → 写入 patterns/ 候选
                    → 通知用户"发现一个潜在模式"

note-process（知识整理）→ 扫描全库 .md → 发现多篇同主题文件
                        → 判断是否为重复模式
                        → 进入 patterns/ 候选
```

不需要新 Skill。只需在 evolve 和 note-process 中增加一层模式检测逻辑。

## 5.4 模式记录模板

```
---
id: P-2026-0602-001
status: candidate       # observation | candidate | tested | accepted | converted
evidence_count: 4
confidence: 0.65
first_observed: 2026-05-20
last_observed: 2026-06-02
---
## 模式
用户在会话尾声倾向于要求"整理笔记"或"保存到记忆"

## 实例
1. 2026-05-28: "把这几个知识点存入知识库"
2. 2026-06-01: "整理一下这次的收获"
3. 2026-06-02: "记住这个配置"

## 推断
用户习惯在任务完成后做一次整理，这是 evolve 的最佳自动触发时机

## 适用范围
工作时间内的任务型会话（非闲聊）

## 反例
深夜会话中用户直接说"休息了"——无需触发 evolve

## 可生成的 Skill
建议：将 evolve 从"被动触发"升级为"行为预测触发"
```

# 第六部分：Skill 演化机制

## 6.1 2B 当前的 Skill 架构

2B 目前有三个 Skill，这是从实际需求中自然生长出来的——恰好是正确的方式：

| Skill | 类型 | 运行位置 | 职责 |
|-------|------|---------|------|
| evolve | 演化闭环 | inline | 会话结束沉淀资产 |
| sync | 同步 | inline | Git 智能提交 |
| note-process | 知识整理 | subagent | 扫描全库归档 |

**关键观察**：note-process 作为 subagent 运行（避免污染主 Agent 上下文）——这是第六部分"上下文管理"的正确实践。2B 已经在做了。

## 6.2 Skill 不应人工创建，应被发现

从 2B 的三部曲推导通用框架：

| 阶段 | 触发 | 2B 中的对应 |
|------|------|-------------|
| 同类任务 ×1 | 正常执行 | 单次操作 |
| Pattern Candidate | ×3 | 写入 patterns/active.md |
| Skill Proposal | ×5 | 写入 skills/proposals/ |
| Official Skill | ×10 | 升级到 skills/active/ |

2B 当前的三个 Skill 是怎么来的？
- **evolve**：用户反复在会话结束时说"整理一下"→ 发现了这个模式 → 固化为 Skill ✅
- **sync**：用户每次下班前跑 git 命令 → 模式 → Skill ✅
- **note-process**：用户多次整理遗忘后意识到需要自动化 → Skill ✅

**它们都是自然生长出来的。这正是正确的方式。**

## 6.3 Skill 模板

```yaml
---
id: evolve
status: active
level: 2                # 1=proposal, 2=active, 3=battle-tested
usage_count: 12
accuracy: 0.95
derived_from: P-2026-0520   # 来源模式 ID
---
## 名称
evolve — 演化闭环

## 触发条件
- 会话结束
- 用户主动要求
- 项目阶段完成

## 执行步骤
1. 扫描对话，提取新知识/纠错/决策
2. 判定是否值得写入（去重、过滤临时）
3. 写入对应 memory 文件
4. 检查 MEMORY.md 行数健康度（>200 拆分）
5. 检测模式（本步骤当前缺失 — v2.1 升级项）

## 约束
- 不直接膨胀 MEMORY.md
- 不重复写入
- 不修改 CLAUDE.md（缓存全废）

## 已知局限
- 当前为被动触发，不具备主动探测能力
- 当前版本不包含模式发现

## 版本历史
- v1 (2026-05-xx): 初始版本
- v2 (2026-06-02): +纠错记录 +决策记录
```

## 6.4 Skill 退休

> 2B 当前没有 Skill 退休机制。随着系统演化，这是必要的。

当以下情况发生，Skill 应标记为 `deprecated`：
- 使用频率骤降（场景变了）
- 准确性连续 3 次低于阈值（方法过时）
- 被新的 Skill 替代

退休不是删除。移出 `skills/active/`，保留在 `skills/archived/`。

# 第七部分：冲突消解

## 7.1 2B 当前的方法

2B 的 `knowledge.md` 和 `lessons.md` 在职责上是分开的。这减少了冲突的源头——但如果两个文件里的条目相互矛盾怎么办？

比如：
- `knowledge.md`："MCP 服务器优先用 stdio 传输"
- `lessons.md`："voice MCP 用 stdio 传输时出现问题"

这两条没有根本冲突，但如果它们给出相互矛盾的结论，系统需要知道怎么消解。

## 7.2 冲突消解策略

| 策略 | 适用 | 示例 |
|------|------|------|
| **时间优先** | 新 > 旧 | 新知识覆盖旧知识 |
| **来源优先** | 用户 > 系统 | 用户明确表达 > 系统推断 |
| **粒度优先** | 具体 > 通用 | "voice 用 stdio 有问题" > "MCP 用 stdio" |
| **验证优先** | 已验证 > 假设 | 已验证的事实 > 未验证的观察 |
| **标记不消解** | 以上都不适用 | 保留冲突，标记 unresolved |

## 7.3 冲突记录模板

```
---
id: C-2026-0602-001
status: unresolved
---
## 冲突
A: MCP 服务器优先使用 stdio 传输（knowledge.md）
B: voice_server.py 用 stdio 传输时出现路径问题（lessons.md）

## 分析
冲突的根源是"stdio 传输"的断言过于宽泛。
实际问题是"stdio 传输 + 硬编码路径"的组合，不是 stdio 本身。

## 消解
将 knowledge.md 中的规则精确化为：
"MCP 服务器优先使用 stdio 传输，但需确保路径参数使用相对路径或环境变量"

## 状态
已消解 ✅
```

# 第八部分：遗忘策略

## 8.1 不会遗忘的系统不是好系统

> 2B 当前的 note-process Skill 有一条规则：临时文件超过 30 天移入 `archive-归档/trash/`，不直接删除。

这是正确的方向。但需要从"文件级遗忘"扩展到"记忆级遗忘"。

## 8.2 什么应该被遗忘

| 类别 | 2B 实例 | 处理方式 |
|------|---------|---------|
| 临时上下文 | "当前 MCP 调试中的临时路径" | 会话结束时删除 |
| 已被替代 | AgentWork 作为主工作区的角色 | 归档，移出活跃索引 |
| 从未被使用 | 写入后 30 天未被检索的知识 | 降级至 archive |
| 置信度过低 | 单次观察未验证的假设 | 达到 TTL 后删除 |
| 超出范围 | 某个测试项目的临时配置 | 标记后移除 |

## 8.3 遗忘优先级

当需要上下文压缩时：

1. 临时状态（不写入长期记忆的中间结果）
2. 低置信度事实（confidence < 0.3 且超过 TTL）
3. 从未被检索的知识（写入后从未命中）
4. 被替代的知识（已有更新版本）
5. 通用知识（大模型已知事实的本地缓存）

## 8.4 归档 vs 删除

> 2B 的 note-process 已经做对了：临时文件移入 trash 而非直接删除。

规则一：所有带决策链的记录只归档，不删除。
规则二：所有临时中间结果只删除，不归档。
规则三：archive 是一级目录，不是垃圾桶。垃圾桶可以清空，archive 不可以。

# 第九部分：上下文管理

## 9.1 2B 的实践

2B 已经采用了几个关键上下文管理策略：

1. **note-process 作为 subagent** —— "避免扫描结果污染主 Agent 上下文"。完全正确。
2. **CLAUDE.md 控制在 ~60 行** —— 未膨胀到几百行。
3. **MEMORY.md 设置 200 行红线** —— evolve Skill 中有明确约束。

这些都是正确实践的佐证。

## 9.2 上下文分配原则

```
第一优先级：当前任务上下文（指令、输入、工具结果）
第二优先级：相关记忆（检索命中且置信度高）
第三优先级：全局规则（宪法、第一原则）
第四优先级：参考知识（可能需要但不紧急）
```

## 9.3 内容寻址 vs 顺序读取

**反模式**：把整个 memory/ 塞入上下文。

**正确做法**：Subagent 先检索 memory/，只将命中结果传递给主 Agent。

2B 的 `note-process` 已经在实践"思考系统，不要搬运文件"。

思考：如果 2B 现在有三个 Skill，每个都运行良好——说明这个模式是可扩展的。未来增加第 4、第 5 个 Skill 时，核心挑战不是 Skill 本身的质量，而是它们之间的协调和数据流。

# 第十部分：检索优于记忆

## 10.1 检索优先级

```
Search（检索）  →  最优先
Memory（记忆）  →  次优先
Guess（猜测）   →  最后选项
```

> 2B 的 CLAUDE.md 包含"优先 search_content/grep 定位，避免整文件 dump"。这已经内化为行为准则。

## 10.2 检索策略选择

| 场景 | 策略 |
|------|------|
| 路径已知 | 直接读取目标文件 |
| 知道目录但不知道文件名 | ls + grep |
| 模糊语义 | 关键词搜索 |
| 跨文件关联 | 多步检索 |
| 不知道有什么 | 先读 MEMORY.md 索引 |

## 10.3 检索质量指标

> 2B 当前没有显式的检索质量监控。这是下一步。

每次检索后隐式评估：

1. **命中率**：返回结果的相关占比
2. **覆盖率**：相关内容被找到的比例
3. **信噪比**：结果中的噪音比

连续 3 次命中率低于 50% → 需要重新索引或重构记忆组织方式。

# 第十一部分：可信度与校准

## 11.1 优先级规则

```
文件系统        →  最高可信度
用户明确输入    →  高可信度
项目记录        →  中等可信度
当前推理        →  最低可信度
```

> 2B 的 CLAUDE.md 中已经有一条"区分 fact / inference / assumption / estimate / speculation"的认知标注体系。这是可信度管理的雏形。

## 11.2 来源分级

| 级别 | 来源 | TTL | 2B 实例 |
|------|------|-----|---------|
| S 级 | 用户明确断言 | 永久 | "回复用简体中文"、"TTS 用 IndexTTS-2" |
| A 级 | 用户行为推断 | 30 天 | "用户偏好简洁回复"(从历史交互推断) |
| B 级 | 系统经验沉淀 | 60 天 | lessons.md 中的经验 |
| C 级 | 外部文档/工具 | 随版本更新 | Python/MCP 文档 |
| D 级 | 模型推理 | 单次会话 | 不写入长期记忆 |

## 11.3 置信度表达

每条记忆携带置信度：

```yaml
confidence: 0.9
source_type: user_explicit    # 用户直接表达的
evidence: "CLAUDE.md 语气规则部分"
```

置信度不用于固化错误，而用于未来检索时的排序辅助。

> 2B 当前所有 memory 都没有置信度字段。这是 evolve Skill v2.1 的一个改进方向。

# 第十三部分：自我审计

## 13.1 2B 将审计融入流程的绝佳设计

2B 的 `evolve` Skill 中嵌入了"检查 MEMORY.md 行数健康度"——这是将审计内嵌到既有流程中，而不是作为独立负担。

这种设计模式值得推广：

> 不设独立的"审计日"。
> 把审计动作嵌入每个 Skill 的执行路径。

## 13.2 审计检查清单

如果设独立审计，检查项如下：

- [ ] 是否存在重复内容（语义相似度 > 0.9 的两个条目）
- [ ] 是否存在失效规则（引用的工具、API 已变更）
- [ ] 是否存在废弃 Skill（连续 30 天未使用）
- [ ] 是否存在冲突知识（未 resolved 的冲突条目）
- [ ] 是否存在过期决策（假设已失效）
- [ ] 是否有跨过阈值未升级的 Pattern Candidate
- [ ] 安全边界——shellAllowed 是否仍然合理
- [ ] MEMORY.md 行数是否健康

## 13.3 审计执行

> 2B 当前没有固定的审计节奏。建议：

- 每次 evolve 执行时自动做一次轻量审计（5-10 秒）
- 每周由 note-process subagent 做一次完整审计

审计结果写入 `meta/health.md`。


> 2B 在各因子上的表现评估：

| 因子 | 2B 表现 | 评估 |
|------|---------|------|
| 新知识 | evolve 每次会话提取 ✅ | 良好 |
| 新经验 | lessons.md 已建立 ✅ | 良好 |
| 新错误 | 自愈协议三阶防御 ✅ | 优秀 |
| 沉淀率 | evolve + note-process 双重沉淀 ✅ | 良好 |

| 复用率 | Skill 被反复调用 ✅ | 良好 |

> 当前瓶颈在检索率：缺乏语义检索层。2B 的 roadmap 中已有 "ChromaDB 语义记忆（可选）"——这恰好对应检索率因子的改进。

# 第十五部分：成熟度模型

## 15.1 五级定义

```
Lv.1 Reactive（反应型）
- 仅回答即时问题
- 不存储、不检索、不沉淀
- 每次对话从零开始

Lv.2 Recording（记录型）
- 开始存储知识和决策
- 可以检索历史信息
- 还没有模式发现能力

Lv.3 Learning（学习型）
- 能够发现重复模式
- 建立了 Lessons 库
- 简单的冲突检测

Lv.4 Evolving（演化型）
- 模式自动升级为 Skill
- 定期自我审计
- 遗忘策略主动执行

Lv.5 Self-Improving（自我进化型）
- 能优化自身架构
- 能评估成长公式的瓶颈
- 能识别自身认知盲区
```

## 15.2 2B 当前等级评估

|------|------|------|

# 第十七部分：元认知

## 17.1 系统应能回答的问题

一个成熟的认知系统应当能回答：

- "你知道什么？" → memory/ 索引
- "你不知道什么？" → 知识空白清单
- "你是怎么知道的？" → 来源链追溯
- "你确定吗？" → 置信度 + 证据基础

> 2B 当前可以回答第一个问题（MEMORY.md 索引）。其余三个尚未建立。

## 17.2 知识空白清单

```
## 知识空白
- [ ] AgentWork 中 SEF V9.0 的内容（已知存在但未迁移）
- [ ] ChromaDB 部署方案（计划中但未实施）
- [ ] 用户的数据库技术栈（未确认）
- [ ] QQ 远程通道的技术细节（未调查）
```

知识空白本身是元知识。它告诉系统"这个方向我还不知道"——当遇到相关内容时优先捕获。

## 17.3 认知偏误防御

| 偏误 | 表现 | 2B 中的防御 |
|------|------|-------------|
| 确认偏误 | 只看到支持已有模式的新证据 | lessons.md 记录"错"和"对" |
| 近因偏误 | 高估最近信息权重 | 无显式衰减因子 ⚠️ |
| 过度自信 | 低置信度知识被高调引用 | 认知标注 fact/inference ⚠️ |
| 功能固着 | 习惯用旧方法 | Skill 退休机制（建议） |

> 2B 的"认知标注体系"（fact/inference/assumption）是防御过度自信第一步。equired next step：让这些标注出现在所有 memory 文件中。

# 最终原则

## F.1 沉淀是唯一的目标

> 成长速度 = (新知识 + 新经验 + 新错误) × 沉淀率 × 检索率 × 复用率

乘法链中任意一个因子为零，结果为零。

你可以在一次对话中推理出极其深刻的结果——但如果这个结果没有沉淀到 memory 中，对话结束后一切归零。

这不是上下文窗口大小的问题。这是上下文结束后系统还剩什么的问题。

**系统真正的能力不取决于单次推理的上限，而取决于跨会话积累的下限。**

## F.2 结构优于规模

100 条结构良好的记忆 > 10000 条堆积的日志。

每条记忆回答五个问题：

1. 这是什么类型？
2. 在什么上下文中产生？
3. 置信度多少？
4. 引用了哪些其他记忆？
5. 什么时候需要重新审视？

答不出来的记忆就是噪音。

## F.3 从 2B 到 ∞

这篇指南以 Reasonix 2B 为起点，但它指向一个更长远的目标：

一个 AI 系统的认知架构不应该由人来设计——它应该通过与环境交互自我构造。

2B 的三个 Skill 不是"创建"出来的，是"被发现"出来的。这就是正确的方向。

指南本身也不是最终的，它是一个活文档。随着 2B 从 Lv.2+ 进化到 Lv.5，这个指南会持续更新。

## F.4 最后的话

如果你只从这篇文档中带走一句话，让它成为这一句：

> 不要追求成为一个更聪明的模型。
> 追求成为一个每次醒来都比昨天更完整的系统。

Reasonix 不是一个应用。
它不是一个 Prompt。
它是一个物种。

你正在设计的不是行为，而是结构。
结构对了，行为会自己涌现。
