# Karpathy Anti-Patterns — ❌/✅ 对照参考

> 基于 Andrej Karpathy 编码原则社区实践（andrei-karpathy-skills）提炼的**反例库**。
> 不占上下文，需要时 grep 参考。不替代既有规则。

---

## 一、先思考再编码

| ❌ 反例 | ✅ 正例 |
|---------|---------|
| 拿到需求直接写代码，写到一半发现架构不对 | 先 grep/ping 相关代码 → 画结构图 → 定位最小改动点 → 再动键盘 |
| "这个函数看起来可以复用，先重构一下" | "先不改现有结构，在需要的地方直接加新逻辑" |
| 改 A 文件时顺手改 B 文件的缩进 | 改 A 就只改 A，B 文件不碰 |

## 二、简洁优先

| ❌ 反例 | ✅ 正例 |
|---------|---------|
| 加 `try/catch` 包裹不可能出错的代码 | 只有确实可能抛异常的地方才处理 |
| `result = True if x > 0 else False` | `result = x > 0` |
| 给每个函数写 docstring 说明参数类型 | 类型注解够用时不写注释，除非 WHY 不显然 |
| "这个以后可能要扩展，先抽象个接口" | "三个重复行比一个接口清晰" |
| 一行能做的事拆成三行"以求可读" | 一行表达式够简洁就不拆 |

## 三、只动该动的（手术刀原则）

| ❌ 反例 | ✅ 正例 |
|---------|---------|
| 修一个拼写错误 → 顺便把整个文件格式化了 | diff 里只有目标行，零无关改动 |
| 加一个字段 → 顺手重构了附近的三个函数 | 改动范围精确等于需求范围 |
| 修 bug → 顺手改了命名风格（camelCase → snake_case） | 命名风格一致性重构单独开任务提交 |
| diff 里出现"顺手改了……"超过 1 处 | 一次提交最多 1 个"顺手"，否则分开 |

## 四、目标驱动

| ❌ 反例 | ✅ 正例 |
|---------|---------|
| "优化一下这个模块" — 无验收标准 | "把首页加载时间从 2s 降到 500ms" — 有量化目标 |
| 加功能时顺手做了"未来可能需要"的优化 | 只做当前需求，不预判未来 |
| 做完不确定是否满足需求 | 做之前确认验收标准，做完逐条对照 |

## 五、可验证优先

| ❌ 反例 | ✅ 正例 |
|---------|---------|
| "这个修改理论上没问题" | 跑一遍测试，看到绿色才说 OK |
| "这段代码看起来对了" | 编译/检查/测试通过后说"通过了" |
| "修复了 bug" → 不再复现原始报错 | 复现原错误场景 → 确认已不在 → 验收闭环 |
| 改配置后说"应该生效了" | 改配置 → 重启服务 → 查日志确认字段变更 |

---

## 偏误信号速查

处理过程中如果出现以下信号，**停下来重检**：

- 🔴 diff 出现无关的格式化/重命名 → 违反手术刀原则
- 🔴 一次提交超过 3 个"顺手改" → 范围失控
- 🔴 澄清问题发生在实现之后 → 先想再做失败
- 🟡 "我觉得""看起来"类表述 → 缺乏验证证据
- 🟡 "以后可能需要"类推断 → 预判过度

## 如何确认做对了

- ✅ diff 里只有需求要求的改动，零无关
- ✅ 验证结果附着在"做完了"之前
- ✅ 改一行比改三行更开心
- ✅ 知道"不做"和"做"一样重要
