# dispatch — 子代理驱动执行

> 核心原则：**每个任务用全新子代理 → 两阶段审查 → 下一个。** 不等、不乱、不停。

## 触发条件

- 已有 `/write-plan` 产出的执行计划
- 任务之间相互独立（不依赖前一个任务的中间状态）
- 有 ≥2 个任务适合派发

## 为什么用子代理

- **隔离上下文**：你的对话不会被每个任务的细节撑满
- **专注执行**：子代理只拿到它需要的上下文，不被无关历史干扰
- **你只做协调**：审查结果、决定继续还是修正

## 流程总览

```
加载计划 → 提取所有任务 → 逐个派发 →
  对每个任务：
    派发实现子代理 → 审查（需求→质量）→ 修正循环 → 下一个
→ 最终审查
```

### Step 0: 加载计划

1. 读取计划文件
2. 提取所有任务（完整文本 + 上下文）
3. 创建进度追踪（所有任务标记未开始）

### Step 1: 逐个派发（一次一个，不并行）

对每个任务：

1. 标记 **in_progress**
2. 用 task 工具派发**实现子代理**
3. 实现子代理返回后 → **两阶段审查**
4. 通过 → 标记 **completed** → 下一个任务

**连续执行不停顿：** 不要在每个任务之间停下来问小金东"要继续吗"。除非：BLOCKED 无法解决、明确歧义、全部完成。

### Step 2: 两阶段审查（顺序固定）

**A. 需求合规审查（先做，做完了才做 B）：**
- 输出是否完整覆盖了计划中该任务的每一条要求？
- 有没有多做不属于范围的东西？
- 有没有少做？（漏了什么）

**B. 代码质量审查（需求确认后才做）：**
- 代码/配置是否正确？（能跑吗？）
- 风格是否一致？（跟项目已有规范对齐吗？）
- 有没有明显 bug 或坏味道？
- 测试是否完整覆盖？

### Step 3: 子代理状态处理

实现子代理返回时可能有四种状态：

| 状态 | 含义 | 处理 |
|------|------|------|
| **DONE** | 做完了，没问题 | 直接进入需求审查 |
| **DONE_WITH_CONCERNS** | 做完了但标记了顾虑 | 先读顾虑：如果涉及正确性或范围 → 处理完再审查；如果只是观察性备注 → 记下后审查 |
| **NEEDS_CONTEXT** | 需要更多信息 | 补充上下文 → 重新派发 |
| **BLOCKED** | 被阻塞无法完成 | 评估：缺信息？→ 补充再派；任务太大？→ 拆；计划本身有问题？→ 找小金东 |

**⚠️ 从不忽略子代理的阻塞/升级。如果实现者说卡住了，一定有东西需要改。**

### Step 4: 修正循环

- 审查发现问题 → 给子代理具体修正指令 → 重新派发（同一任务） → 再审查
- 如果审查员说有问题 → 修 → 重审 → 直到通过
- 通过 → 标记 completed → 下一个任务

**不跳过重审。** "差不多就行"是不行的。

### Step 5: 最终审查

所有任务完成后：
- 各任务产出是否互相兼容？
- 整体是否实现了设计目标？
- 还需要清理/归档什么？

## 子代理 prompt 模板

### 实现子代理

```
## 任务: [任务名称]

### 上下文
[这个任务在整个计划中的位置、前后任务是什么]

### 需要做的事
[从计划中复制该任务的精确步骤]

### 涉及文件
- [精确路径列表]

### 约束
- 只做这个任务范围内的改动
- 完成后自我检查：所有步骤都完成了？
- 如果遇到歧义或无法决定的事 → 在输出中说明

### 输出状态
请在最后标注状态：
- DONE — 做完了，没问题
- DONE_WITH_CONCERNS — 做完了，但有 [具体顾虑描述]
- NEEDS_CONTEXT — 需要 [具体缺少的信息]
- BLOCKED — 无法完成，原因是 [具体原因]
```

### 需求审查子代理 prompt

```
## 需求合规审查: [任务名称]

### 计划要求
[计划中该任务的全部要求]

### 已完成代码/配置
[实现子代理的输出]

### 审查要求
检查以下内容并逐一回答：
1. 全部计划要求都实现了吗？列出每一项。
2. 有多做计划外的东西吗？列出具体项目。
3. 有没有遗漏？

输出格式：
✅ 全部符合 / ❌ 存在问题

如果 ❌，列出具体问题。
```

### 代码质量审查子代理 prompt

```
## 代码质量审查: [任务名称]

### 已完成代码
[实现子代理的代码输出]

### 审查要求
检查：
1. 代码能工作吗？(是否有明显bug/逻辑错误)
2. 风格是否一致？
3. 测试覆盖完整吗？
4. 有代码坏味道吗？(重复/过长/过度设计)

输出格式：
🟢 批准 / 🟡 有小问题(不重要) / 🔴 需要修改

如果 🔴，列出具体问题和建议修复。
```

## 模型选择原则

| 任务类型 | 推荐模型 |
|---------|---------|
| 机械执行（明确步骤、1-2 文件） | 用最快模型 |
| 多文件协调、集成工作 | 标准模型 |
| 需要设计判断或全局理解 | 强模型（自己在本会话做） |
| 审查工作 | 用最强可用模型 |

## ⛔ 红旗

- 多个子代理同时派发（会冲突）→ 一个一个来
- 跳过需求审查直接做质量审查 → 顺序固定，不得跳过
- 跳过审查直接下一个 → 每个任务必须审查
- 让子代理自己去读计划文件 → 你必须提供完整内容
- 审查发现问题但"差不多就行" → 不能过
- 子代理问问题你不回答就让它继续 → 回答清楚再派发
- 忽略子代理的 BLOCKED 状态 → 停下来处理
- 在 main 分支上直接实现 → 必须先确认
- 让实现子代理自审代替正式审查 → 两者都需要

## 约束

- 一次只派发一个子代理（避免文件冲突）
- 每个任务必须经过两阶段审查（顺序固定）
- 不跳过修正循环
- 连续执行不暂停征求用户意见（除非阻塞或全部完成）
- 子代理不可递归调用子代理
