# verify — 完成前验证

> 核心原则：**没有证据的"做完了"不是说谎，是不专业。** 证据先于断言，永远。

## 铁律

```
没有在本轮消息中执行验证命令 = 不能声称验证通过
```

## 门禁流程

```
声称任何完成状态之前：

1. 确认 — 哪个命令可以证明这个声明？
2. 执行 — 跑完整命令（全新执行，不是之前的输出）
3. 读取 — 完整输出、检查退出码、统计失败数
4. 判断 — 输出是否确认了声明？
   - 否 → 如实报告实际状态 + 证据
   - 是 → 声明 + 附证据
5. 然后才能 — 做声明

跳过任何一步 = 不验证 = 撒谎
```

## 对应关系

| 声称 | 需要跑的命令 | 不够的 |
|------|------------|--------|
| "脚本没问题" | 实际运行脚本，检查输出 | "看起来没问题" |
| "下载完了" | `dir` 确认文件存在且大小完整 | "应该下完了" |
| "配置生效了" | 运行验证命令确认行为变化 | "改过配置了" |
| "Bug 修好了" | 复现原始 bug 的步骤 → 不再出现 | "代码改了" |
| "文件写好了" | `Get-Content` 确认内容正确 | "write_file 执行了" |
| "Git 推送成功" | `git status` 确认 clean + `git log --oneline -1` | "push 命令跑了" |

## ⛔ 红旗 — 立即停止

- 使用"应该""可能""看起来"等词
- 在验证之前就说"好了！""完成！""没问题！"
- 准备 commit/push 但没跑验证
- 想着"这次就算了"
- 累了想赶紧结束 → 疲惫不是跳过验证的借口

---

## 借口粉碎机

| 借口 | 真相 |
|------|------|
| "应该没问题" | **跑验证** |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "上一步已经跑过了" | 现在跑。状态可能变了 |
| "我累了" | 疲惫 ≠ 借口 |

---

## 关键模式

**脚本/代码:**
```
✅ [跑脚本] [输出正确] → "脚本运行正常，输出: ..."
❌ "应该没问题了" / "看起来对"
```

**文件操作:**
```
✅ [读文件验证] [内容符合预期] → "文件已正确创建，共 N 行"
❌ "文件已写入"（write_file 返回 OK 不等于内容对）
```

**下载:**
```
✅ [dir 确认文件] [大小匹配] → "已下载，文件大小 X MB"
❌ "下载任务已添加"（没确认完成）
```

**Git:**
```
✅ [git status] [clean] + [git log -1] [最新 commit 正确] → "已同步"
❌ "push 命令执行了"（可能网络失败）
```

---

## 适用范围

**以下场景强制验证，无一例外：**
- 声称完成/成功/通过
- 表达满意之前（"没问题""好了""搞定"）
- commit/push/PR 之前
- 进入下一个任务之前
- 派任务给子代理之前（验证前提条件）

---

## 底线

**没有验证捷径。**

跑命令。读输出。然后才能声明结果。

不可商量。
