# tdd — 测试驱动工作法

> 核心原则：**你没看到它失败，你就不知道它测对了没。** 红 → 绿 → 重构，永远是这个顺序。

## 铁律

```
没有先看到验收标准失败之前，禁止产出实现。
```

先写了实现再补验收标准？**删掉实现。从头来。**
没有例外。不保留"参考"。不"适配"已写代码。删除就是删除，从测试开始重写。

## RED-GREEN-REFACTOR 循环

```
RED    → 定义验收标准，跑它，确认它失败（且失败原因正确）
GREEN  → 做最少的事让它通过，不额外加功能
REFACTOR → 通过后整理，不改变行为，保持通过
```

## 适用范围（不限于代码）

| 产出类型 | "测试"是什么 |
|---------|------------|
| 脚本/代码 | 实际运行，检查返回值、输出、副作用 |
| 配置文件 | 运行验证命令，确认行为符合预期 |
| 文档 | 检查清单逐项比对 |
| 数据处理 | 小样本输入 → 验证输出 |
| 下载/安装 | 验证文件存在、大小、版本 |

## RED — 写失败的验收标准

做一件事之前，先写一个最小验证展示期望行为：

**代码示例：**
```python
# 先写这个，跑它，确认报错（函数还没写）
def test_filter_empty_lines():
    result = filter_lines(["a", "", "b", "", "c"])
    assert result == ["a", "b", "c"]
```

**配置示例：**
```bash
# 先跑这个，确认当前配置不满足预期
curl http://127.0.0.1:9877/ping
# 预期: 连接失败（TTS 服务器还关着）
```

**文档示例：**
```
检查清单:
[ ] 所有路径已验证存在 → 跑 dir 逐路径检查
[ ] 所有命令已验证可执行 → 跑 Get-Command 逐命令检查
```

### RED 验证（必须 — 不可跳过）

```bash
# 跑验收标准，确认它失败
```

确认三点：
- ✅ 测试失败（不是报语法错误）
- ✅ 失败原因符合预期（因为功能缺失，不是其他原因）
- ✅ 失败消息能清楚告诉你"缺什么"

如果测试通过了？→ 你在测已有功能，修正标准。
如果测试报语法错？→ 修正后重跑。
如果测试因其他原因失败？→ 修测试，不是为了通过，而是为了**正确失败**。

## GREEN — 最少实现

只写刚好让标准通过的代码，不额外加功能：

```python
def filter_lines(lines):
    return [l for l in lines if l != ""]
# 不多加 strip、不去空白行、不处理 None
```

- 不加"以后可能用到"的功能 (YAGNI)
- 不重构其他代码
- 不"顺便改进"

### GREEN 验证（必须 — 不可跳过）

```bash
# 跑验收标准
# 确认：通过，且无其他标准被破坏
```

---

## REFACTOR — 整理

只在全部通过后：
- 去重
- 改名
- 提取公共逻辑
- 保持所有标准通过

## 完整示例：Bug 修复全流程

**Bug：** 空邮箱地址被系统接受。

### RED
```python
def test_rejects_empty_email():
    result = submit_form({"email": ""})
    assert result.error == "Email required"
```

### 验证 RED
```bash
pytest tests/test_form.py -v
# 预期：FAIL — expected 'Email required', got None
```

### GREEN
```python
def submit_form(data):
    if not data.get("email", "").strip():
        return {"error": "Email required"}
    # ... rest of the logic
```

### 验证 GREEN
```bash
pytest tests/test_form.py -v
# 预期：PASS
```

### REFACTOR
如果多个字段需要验证，提取公共验证逻辑。保持测试通过。

## 为什么这个顺序不能颠倒

**"我先写实现再补验收标准"**

实现之后写的标准跑一次就通过。立刻通过什么也证明不了：
- 可能测的是实现而非需求
- 可能漏了边界条件
- 你从未见过它抓住 bug

**"写完再写测试也能达到同样目的"**

错。**写后测试回答"代码做了什么"；写前测试回答"代码应该做什么"。**
写后测试被你的实现偏差影响——你测你做的，不测你漏的。
写前测试迫使你在实现之前发现边界情况。30分钟的补测试 ≠ TDD。你有覆盖率，但没有测试真的在工作。

**"我已经手动测过了"**

- 手动测试没有记录，无法复现
- 代码改了没法重跑
- 你以为测了全部但实际漏了
- "我试过可以" ≠ 全覆盖

**"删掉N小时的实现太浪费了"**

沉没成本谬论。时间已经花掉了。你的选择是：
- 删掉用 TDD 重写（再花N小时，高确信度）
- 留着然后补测试（30分钟，低确信度，大概率有bug）

"浪费"是留着你不信任的代码。没经过真测试的能跑的代码 = 技术债务。

**"TDD太教条了，务实就是要灵活"**

TDD 本身就是务实的：
- 提交前发现 bug（比上线后调试快得多）
- 防止回归（测试立刻抓住破坏）
- 代码自我文档化（测试展示怎么用）
- 可以放心重构（测试抓住所有破坏）

"务实"的偷懒 = 在上线后调试 = 更慢。

## 常见借口粉碎机

| 借口 | 真相 |
|------|------|
| "太简单了不用测" | 简单代码也会坏。测试只要30秒。 |
| "我之后补测试" | 之后补的测试一次通过——证明不了什么。 |
| "写完再写测试也能达到同样目的" | 写后测 = "代码做了什么"；写前测 = "代码应该做什么" |
| "已经手动测过了" | 手动 ≠ 系统化。没记录、不能重跑。 |
| "删了N小时太浪费" | 沉没成本谬论。留着不信任的代码才是浪费。 |
| "保留已写代码当参考，从测试开始" | 你一定会改编它，那就是写后测了。删掉就是删掉。 |
| "需要先探索一下" | 可以。探索完丢掉，从 TDD 开始。 |
| "测试难写 = 设计不清楚" | 听测试的话。难测 = 难用。 |
| "TDD会拖慢我" | TDD 比调试 bug 快。务实 = 测试优先。 |
| "已有代码没有测试" | 你现在在改进它。先给旧代码加测试，再改。 |

## ⛔ 红旗 — 遇到立刻停止，从头来

- 实现写在测试之前 → 删实现，重来
- 测试第一次就跑通了 → 测试有问题（你在测已有功能）
- 说不清测试为什么失败 → 没理解测试在测什么
- "之后补测试" → 不补，现在补
- "太简单不需要" → 最简单的代码最需要测试
- "只这一次跳过" → 没有例外
- "我已经手动测过了" → 手动测试 ≠ 自动化测试
- "保留已写代码当参考" → 那就是写后测

**以上任何一条出现 = 删代码。从 TDD 开始。**

## 完成验证清单

在把工作标记为完成之前确认：
- [ ] 每个新函数/方法都有测试
- [ ] 每个测试都在实现之前看到它失败
- [ ] 每个测试失败的原因是对的（功能缺失，不是语法错）
- [ ] 写了最少代码让每个测试通过
- [ ] 所有测试通过
- [ ] 输出干净（没有错误、警告）
- [ ] 测试用真实代码（mock 只在不得已时用）
- [ ] 边界情况和错误被覆盖了

无法勾完所有？→ 说明跳过了 TDD。**从头来。**

## 约束

- 每次改动都走 RED → GREEN → REFACTOR
- 严禁跳过 RED 验证步骤
- TDD 不是教条，是务实——它比调试 bug 更快
- 没有小金东的批准，绝不跳过 TDD
