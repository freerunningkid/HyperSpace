# debug — 系统排错

> 核心原则：**绝不猜错。** 随机修复浪费时间且制造新 bug。先找根因，再动手。

## 铁律

```
未完成 Phase 1（根因调查）之前，禁止提出任何修复方案。
Phase 1 完成之前的所有修复提议都是猜测。猜测是 bug 的源头。
```

## 四阶段流程

### Phase 1: 根因调查

**在尝试任何修复之前，必须完成：**

1. **仔细读错误信息** — 不要跳过任何报错/警告，它们往往包含精确答案。读三遍。
2. **稳定复现** — 能可靠触发吗？确切的步骤是什么？每次必现？偶发？
3. **检查最近变更** — `git diff`、最近 commit、新增依赖、配置变更
4. **多组件系统：组件边界诊断**

   ```
   在每个组件边界：记录入参 → 记录出参 → 验证状态传递
   跑一次收集证据 → 定位哪个组件出问题 → 只调查那个组件
   ```

5. **追踪数据流**（深层错误时）：

   - 错误值从哪里来？
   - 谁传入了这个错误值？
   - 沿调用链向上追到源头
   - **在源头修，不在症状上修**

6. **防御性调试**（偶发/条件性 bug）：

   ```
   在怀疑的组件边界加日志/断言 → 复现 → 读日志 → 定位
   每次只加一个诊断点，不要全部加完一起跑
   ```

7. **条件等待法**（竞态条件/时序 bug）：

   ```
   不要用 sleep() 等固定时间 → 改为轮询条件直到满足
   "等 5 秒" 在慢机器上不够，在快机器上浪费
   用: wait_until(condition, timeout=10)
   ```

### Phase 2: 模式分析

1. **找正常工作的同类** — 本项目中有没有类似但正常工作的代码/配置？
2. **对比差异** — 列出正常 vs 异常的**每一个差异**，别假定"这个不可能有关系"
3. **理解依赖** — 这个组件依赖什么？什么配置？什么环境变量？

### Phase 3: 假设和验证

1. **形成单一假设** — "我认为根因是 X，因为 Y"
2. **最小化测试** — 只改一个变量来验证
3. **验证后再继续** — 成了 → Phase 4；没成 → 新假设，别堆更多修复

**多个假设 = 不知道哪个是真的。** 一次验证一个。

### Phase 4: 实施修复

1. **写复现用例** — 最简单能触发 bug 的脚本/步骤（修复前必须有）
2. **实施单一修复** — 只修根因，不做"顺便改一下"
3. **验证修复** — 用例通过？没有引入新问题？
4. **如果修复失败**：
   - 尝试 <3 次 → 回到 Phase 1 重新分析
   - **尝试 ≥3 次 → 停止！质疑架构本身**："这个方案本身对吗？是不是在用惯性硬撑？"
   - 3 次修复都失败通常意味着：不是 bug，是设计问题

## ⛔ 红旗 — 以下行为立刻停止，回 Phase 1

- "先试试改 X 看行不行"
- "一次性改多处跑一下"
- "应该是 X 的问题，让我修一下"
- "不太确定但先试试"
- "再试一次"（已经试了 2 次以上）
- 每个修复都暴露新问题 → 这是架构问题，不是 bug
- "我知道问题在哪让我直接改" → 看出症状 ≠ 理解根因

## 常见借口粉碎机

| 借口 | 真相 |
|------|------|
| "问题简单不需要流程" | 简单问题也有根因。流程对简单 bug 更快 |
| "紧急没时间走流程" | 系统排错比瞎猜-撞墙循环**更快** |
| "先试这个再调查" | 第一次修复定基调。从一开始就做对 |
| "我看出问题了让我修" | 看出症状 ≠ 理解根因 |
| "再试一次说不定就好了" | 同样的方法期待不同结果 = 疯狂 |
| "这个 bug 是偶发的没办法" | 偶发 bug 更需要系统方法。加诊断→条件等待法 |

## 防御性调试实战

**场景：** 一个多步骤流水线偶尔在第 3 步失败。

**错误做法：** 猜"可能第 3 步有问题"→ 看两眼 → 觉得没问题 → 改别的 → 再试。

**正确做法：**
1. 在每个步骤开头加日志：`[Step N] input=<关键参数摘要>`
2. 复现 → 看日志 → 定位具体到哪个步骤、哪个参数
3. 在那个步骤加更细的日志：入参 → 内部计算 → 出参
4. 定位到具体行
5. 修复
6. 去掉临时日志

**核心：每次只加一个诊断点，不要一次性加满。从宏观到微观。**

## 速查表

| 阶段 | 核心动作 | 完成标准 |
|------|---------|---------|
| 1. 根因 | 读错误、复现、查变更、加诊断 | 理解 WHAT 和 WHY |
| 2. 模式 | 找正常同类、对比差异 | 定位差异点 |
| 3. 假设 | 形成理论、最小测试 | 确认或新假设 |
| 4. 修复 | 写复现用例、修、验证 | Bug 消失、无新问题 |

## 约束

- 修复前必有复现用例
- 3 次修复失败 → 停止，质疑架构，跟小金东讨论
- 绝不在没理解根因的情况下动手
- 临时诊断日志修复完成后必须清理
