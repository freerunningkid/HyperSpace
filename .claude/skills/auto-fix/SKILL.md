# auto-fix — 自动修复循环

> 定位：**子代理（subagent）**
> 解决：我（主 Agent）规划→执行→发现失败→修复的循环占用上下文，auto-fix 一次性闭环

## 用法

主 Agent 调用：
```
run_skill("auto-fix", "<任务描述>")
```

任务描述格式：
```
目标: <明确的验收标准>
命令: <初始执行命令>
文件: <涉及的文件路径>
约束: <任何需要保留的限制>
```

## 子代理工作流程

### Step 1：运行初始命令

执行 `命令`，捕获 stdout + stderr + exit code。

### Step 2：判断结果

- ✅ **通过**（exit 0 + 预期输出）→ 结束，返回成功报告
- ❌ **失败** → 进入修复循环

### Step 3：修复循环（最多 3 轮）

每轮：
1. **诊断** — 分析错误输出，定位根因。从实际输出中读，不猜。
2. **修复** — 修改相关文件，只改最少内容
3. **重试** — 重新运行命令
4. **判断** — 同 Step 2

3 轮后仍未通过 → 进入 Step 4。

### Step 4：输出带状态的报告

报告开头必须包含**状态字段**，帮助主 Agent 判断下一步：

```
══════════════════════════
 AUTO-FIX REPORT
══════════════════════════
状态: DONE / DONE_WITH_CONCERNS / BLOCKED

📋 初始命令: ...

🔄 修复尝试:
  1. [诊断] → [修复内容] → [结果]
  2. ...
  3. ...

📌 最终状态: ...
📝 变更文件: [列表]

💡 建议（BLOCKED 时必填）: ...
══════════════════════════
```

**状态说明：**
| 状态 | 含义 |
|------|------|
| DONE | 成功修复，验证通过 |
| DONE_WITH_CONCERNS | 修复了，但对某部分修复不完全满意（描述顾虑） |
| BLOCKED | 3 轮尝试后无法修复，需要主 Agent 介入 |

## 适用范围

| 场景 | 示例 |
|------|------|
| 脚本/程序运行失败 | `python test.py` 报错 - 读输出→修代码→再跑 |
| 编译错误 | `go build` 失败 - 读错误→修源码→重编译 |
| 测试不通过 | `pytest` 有失败用例 - 读失败信息→修代码→重跑 |
| 部署配置生效失败 | 进程未启动 - 检查配置→修复→重启验证 |

## 铁律

- **不猜测根因** — 必须从命令实际输出中提取错误信息
- **每次修复只改最少内容** — 不要顺带重构
- **子代理不可递归调用子代理** — 修复循环在子代理内部完成
- **不询问用户** — 子代理没有 ask 工具，自行诊断修复
- **无法修复时如实报告** — 不要编造成功
- **状态不可省略** — 报告必须包含状态字段

## 与 verify skill 的关系

- `verify`：在主会话中用，防止我说"做完了"却没跑验证
- `auto-fix`：在子代理中用，自动执行"跑→测→修→跑"闭环
- 配合使用：主 Agent 规划后 → `dispatch` 派任务给子代理 → 子代理内 `auto-fix` 自动修复
