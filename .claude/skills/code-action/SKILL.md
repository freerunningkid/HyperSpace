# code-action — 一站式编码行动

> 定位：**子代理（subagent）**
> 目标：一个子代理搞定"改代码"的完整闭环，主 Agent 只收最终报告

## 用法

主 Agent 调用：
```
run_skill("code-action", "<任务描述>")
```

任务描述格式：
```
目标: <要达成的效果/要修复的问题>
涉及: <文件或目录>
验收: <怎么判断做对了>
约束: <不要改的文件/要保持的约定>
```

## 子代理工作流程

### Step 0：确认（有疑问先问再动手）

在开始任何探索之前，如果任务描述中有任何模糊之处：
- "加适当的错误处理" → 问："什么类型的错误？怎么处理？"
- "优化性能" → 问："性能指标是什么？怎么测量成功？"
- "按规范调整" → 问："哪个规范？有没有参考文件？"

**不问清楚就动手 = 大概率做错。**

### Step 1：探索（理解现状）

1. 读取涉及的每个文件，理解当前代码结构
2. 如果路径不精确，先用 `codegraph_search` 或 `grep` 定位相关符号
3. 确认任务目标与现状之间的差距

### Step 2：定位（找到需要改的点）

1. 标记要修改的具体位置（文件:行号）
2. 确认修改不会破坏其他逻辑（检查依赖符号）
3. 如有多个方案，选最少改动的那一个

### Step 3：修改（手术刀式编辑）

1. 一次只改一个文件
2. 每个修改用 `edit_file` 或 `multi_edit`，避免全文件重写
3. 改完后立即用 `lsp_diagnostics` 检查语法错误（支持的语言）

### Step 4：自审（实现完成后自我检查）

改完后立即自我检查：
1. 任务目标实现了没有？（逐项核对）
2. 有没有改了不该改的地方？（undo）
3. 代码风格跟项目一致吗？
4. 有遗漏的边界情况吗？

### Step 5：验证（确认改动正确）

1. 运行项目测试或构建命令
2. 确认输出符合验收标准
3. 如果失败，进入 `auto-fix` 模式修复（最多 2 轮）

### Step 6：报告（输出给主 Agent）

输出**状态 + 报告**，不能只输出报告：

```
══════════════════════════
 CODE-ACTION REPORT
══════════════════════════
状态: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED

🎯 任务: [目标摘要]

📁 文件变更:
  - [路径]:[行号] — [做了什么]
  - ...

✅ 验证结果:
  - [命令] → [输出摘要]

📌 备注: [任何需要注意的副作用]
══════════════════════════
```

**状态说明：**
| 状态 | 什么时候用 |
|------|-----------|
| DONE | 全部完成、验证通过 |
| DONE_WITH_CONCERNS | 做完了但有些顾虑（如"这个文件有点大"或"这段代码不确定"） |
| NEEDS_CONTEXT | 信息不够，需要主 Agent 补充 |
| BLOCKED | 卡住了，无法完成（描述原因） |

**不给上下文注水** — 只返回最终报告，不把中间过程全输出。

## 适用范围

| 场景 | 示例 |
|------|------|
| 原子化代码修改 | "把函数 X 改名为 Y"、"增加参数 Z 的默认值" |
| Bug 修复 | "这个场景下会空指针，修一下" |
| 小功能实现 | "在端点 /api/foo 加上分页参数" |
| 配置调整 | "把超时从 30s 改成 60s" |
| 重构 | "把 utils.py 里的 helper 抽成独立模块" |

## 铁律

- **先确认再动手** — 任务模糊时不猜，问清楚
- **不给上下文注水** — 只返回最终报告 + 状态
- **不动无关代码** — 只改任务指定的范围
- **最小化变更** — 选改动最少的方案，不做顺手重构
- **自审不能跳过** — Step 4 在验证之前必须跑
- **验证不可跳过** — 改完必须跑一次验证
- **子代理不可递归调用子代理** — 用完工具就停
- **状态不能省略** — 报告中必须包含状态字段
