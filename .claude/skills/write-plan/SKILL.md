# write-plan — 执行计划拆解

> 核心原则：**把大象放进冰箱分几步？** 每一个步骤精确到文件路径、命令、预期输出。

## 触发条件

- `/brainstorm` 流程产出设计文档后
- 用户说"拆成任务""写个计划""怎么一步步做"
- 多步骤任务需要追踪进度

**announce**: "我用 write-plan 来拆解执行计划。"

## 流程

### Step 1: 范围检查

如果设计文档涉及多个独立子系统 → 建议拆为多个计划，每个计划应产出可独立测试的成果。
否则继续。

### Step 2: 文件结构映射

任务拆解前，先列出涉及的文件及其职责：
- 哪些新建、哪些修改
- 每个文件的单一职责
- 文件之间的依赖关系

**设计原则**：一起变更的文件应该在一起。按职责分，不按技术层分。
在已有代码库中遵循已有模式。如果正在修改的文件已经变得过于臃肿，合理拆分。

### Step 3: 拆解为 2-5 分钟的任务

每个步骤一个动作的粒度：
- "写失败的测试" — 一步
- "跑测试确认它失败" — 一步
- "写最少实现代码" — 一步
- "跑测试确认通过" — 一步
- "Commit" — 一步

### Step 4: 写计划文档

保存到 `docs/plans/YYYY-MM-DD-<主题>-plan.md`

## 计划文档头

**每个计划必须以这个头开始：**

```markdown
# [功能名称] 执行计划

> **给执行者：** 推荐使用 `/dispatch` (子代理驱动) 或本会话顺序执行来实现此计划。步骤使用 `- [ ]` 语法追踪进度。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句关于方案的关键描述]

**技术栈：** [关键技术和库]

---
```

## 任务格式模板

```markdown
### Task N: [组件名称]

**涉及文件：**
- 新建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py:123-145`
- 测试: `tests/exact/path/to/test.py`

- [ ] **Step 1: 写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **Step 2: 跑测试确认它失败**

```bash
pytest tests/path/test.py::test_name -v
```
预期：FAIL — "function not defined"

- [ ] **Step 3: 写最少实现代码**

```python
def function(input):
    return expected
```

- [ ] **Step 4: 跑测试确认通过**

```bash
pytest tests/path/test.py::test_name -v
```
预期：PASS

- [ ] **Step 5: Commit**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
```

## 零占位符铁律

**以下属于计划失败，绝不允许写在计划里：**
- "TBD" / "TODO" / "稍后补充" / "implement later"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"（没说怎么处理 = 没写）
- "给上面的写测试"（没写具体测试代码）
- "参考 Task N"（把代码抄过来，执行者可能乱序看任务）
- 只描述做什么不展示怎么做（代码步骤必须有具体代码块）
- 引用其他任务中未定义的类型/函数/方法

### 精确到行
- 每个文件路径必须精确
- 每个命令必须完整可复制执行
- 预期输出必须写明（用于验证）

## 自审清单（写完后跑一遍）

1. **设计覆盖**：设计文档的每一条需求都有对应任务吗？列出缺口。
2. **占位符扫描**：搜索任何模糊描述（"适当""合适""类似""稍后"），修复。
3. **类型一致性**：后续任务中用的类型名、方法签名、属性名跟上文的一致吗？

发现问题就修，不需要重新审查。

## 执行交接

计划保存后，给用户选择：

**"计划已保存到 `docs/plans/<文件名>.md`。两种执行方式：**

**1. 子代理驱动（推荐）** — 使用 `/dispatch`，每个任务用独立子代理执行，任务间两阶段审查（需求合规 → 代码质量）。连续执行不停顿。

**2. 本会话顺序执行** — 在当前对话中逐个执行，每批任务后可检查点暂停。

**选哪个？"**

如果选子代理驱动：
- **必须使用 `/dispatch` Skill**
- 每个任务用全新子代理 + 两阶段审查

如果选本会话执行：
- **必须使用本会话的 task 工具**
- 逐个执行，每批后暂停审查

## 约束

- 非平凡多步骤任务必须拆计划，不允许边做边想
- 计划文档必须完整、无占位符
- 计划完成必须经用户确认后才开始执行
